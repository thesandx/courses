PipeForge — Project Charter & Design Document

This is the design doc for the tool you are building. It is deliberately written using the 7-step system-design framework from the notes (§2) — so the very first artifact of the course is you practicing the interview format on your own product. Read it fully before Phase 1; return to it whenever a phase decision feels arbitrary.

0. Vision

PipeForge: an open-source, metadata-driven data platform for GCP. Register a source once; PipeForge generates ingestion, quality gates, SCD2 history, star-schema serving, reconciliation, lineage, auditing, and operations dashboards — with zero per-source code.

Elevator pitch for your README/resume:

PipeForge is a config-driven ETL platform on GCP. It onboards batch and streaming sources through a registration API, lands them in a GCS/BigQuery lakehouse (RAW → ODP → FDP → CDP), and auto-generates Airflow pipelines with built-in data quality (6 dimensions), SCD2 change capture, reconciliation, tokenization of PII, full operational audit, and a Streamlit control plane. Provisioned end-to-end with Terraform.

1. Requirements (Step 1 of the framework)

Functional

  • F1 — Registration: onboard a source system + dataset via CLI/API/UI; store connection info, schedule, load pattern, schema contract, keys, PII flags, DQ rules, SLAs.
  • F2 — Batch ingestion: file drops (CSV/JSON/Parquet from GCS), JDBC tables (full, incremental-watermark), REST APIs (paginated, rate-limited) → RAW zone.
  • F3 — Streaming ingestion: Kafka topics (Avro + Schema Registry) → micro-batched to GCS/BigQuery with DLQ; optional Pub/Sub path.
  • F4 — Processing: RAW→ODP standardization (typing, dedup), ODP→FDP conformance (SCD2, surrogate keys, tokenized PII), FDP→CDP data products (dims/facts/rollups).
  • F5 — Quality: declarative rules across 6 dimensions; actions = pass / warn / quarantine / block; scorecard per dataset.
  • F6 — Reconciliation: count/sum/hash checks source↔layers; divergence alerts.
  • F7 — Orchestration: DAGs generated from metadata; backfill any date range with one command.
  • F8 — Audit & lineage: every run/task captured (rows, bytes, duration, status); dataset-level lineage graph; searchable catalog.
  • F9 — Lifecycle: archival tiers, partition expiry, purge (right-to-delete).
  • F10 — Control plane: FastAPI + Streamlit (catalog search, DQ scorecard, ops console, onboarding wizard).

Non-functional (the 7 DE dimensions, §3)

Dimension Target
Volume Design for 100 GB/day; demo at ~1–5 GB/day (cost) — architecture must not change between the two, only knobs (§8: 10x = engineering)
Freshness Batch: T+1 by 06:00; streaming: < 5 min end-to-end (micro-batch — the pragmatic answer, §27)
Retention RAW 90 days hot → archive; FDP/CDP per registered policy
Reliability At-least-once everywhere + idempotent sinks = effectively-once (§18)
Cost Dev env ≤ $25/month; every table partitioned; serverless-first
Quality 5 baseline checks on every dataset (§8): rowcount, null PKs, freshness, volume anomaly, dup PKs
Compliance PII tokenized before FDP; audit log immutable; purge API

2. Sources & volumes (Step 2)

Demo domain: "NovaMart" — a fictional e-commerce retailer. It gives you every source category from §4 in one coherent story:

Source Category Pattern (§12) Volume (demo) Contents
orders CSV drops Files file landing + trigger + validation 100k rows/day order lines (the classic fact)
customers, products in Cloud SQL (MySQL) Database incremental watermark + periodic full recon; CDC discussion 1–5k changes/day dimensions with updates → SCD2
clickstream Kafka topic Event stream streaming micro-batch 50–500 ev/s from a generator page views, carts, sessions
fx_rates public REST API API scheduled pull, pagination, retries tiny currency enrichment

Back-of-envelope (do this math yourself in Phase 0 — §19): 500 ev/s × 400 B ≈ 17 GB/day raw JSON ≈ 2–3 GB/day Parquet. Fits the free-ish tier; stresses nothing; shape is identical to 100x production.

3. High-level architecture (Step 3)

Six layers left→right (§5), with the enterprise layer names from the reference architecture mapped onto medallion:

SOURCES → INGESTION → RAW (landing) → ODP → FDP → CDP → CONSUMPTION
            ▲                        (bronze) (silver+) (gold)
            └── driven by METADATA CORE (registration, contracts, schedules, rules)
   [ cross-cutting: Workflow · DQ · Recon · Audit · Tokenization · Archival · Lineage/Catalog ]
Layer Store Format Partitioning (§11) Writes
RAW GCS gs://<proj>-pf-raw/ source-native + metadata sidecars source/dataset/ingest_date=YYYY-MM-DD/batch_id= append-only
ODP BigQuery dataset odp (+ GCS Parquet) columnar, typed PARTITION BY ingest_date partition overwrite
FDP BigQuery dataset fdp conformed, SCD2, SKs PARTITION BY business date, CLUSTER BY keys MERGE
CDP BigQuery dataset cdp star schemas, rollups business date + clustering partition overwrite / MV

Why both a GCS lake and BigQuery? RAW must be replayable and engine-agnostic (your backfill insurance, §5/§13); BigQuery is the serving engine. ODP exists in both during the Spark phases (Spark reads/writes GCS Parquet; BigQuery external/BigLake tables expose it) — this teaches the lakehouse pattern (§21) honestly.

4. The metadata model (Step 4) — the heart of the platform

One Cloud SQL Postgres database, schema pipeforge. You will grow it phase by phase; the end state:

source_system(source_id PK, name, kind[db|file|api|stream], connection_ref,  -- Secret Manager key, never creds
              owner_email, tenant, created_at)
dataset(dataset_id PK, source_id FK, name, layer_targets, load_pattern[full|incr|cdc|stream|api],
        schedule_cron, watermark_column, primary_keys[], business_date_column,
        scd_type[1|2], pii_columns[], sla_freshness_minutes, retention_days, active)
schema_contract(contract_id PK, dataset_id FK, version, format[jsonschema|avro],
                definition JSONB, compatibility[backward|forward|full], created_at)
dq_rule(rule_id PK, dataset_id FK, dimension[completeness|accuracy|consistency|
        timeliness|validity|uniqueness], rule_type, params JSONB,
        severity[warn|quarantine|block], active)
pipeline_run(run_id PK, dataset_id FK, execution_date, layer, status, started_at,
             finished_at, rows_read, rows_written, rows_quarantined, bytes_written,
             error, airflow_dag_id, airflow_run_id)          -- the AUDIT MODEL ⑧
dq_result(result_id PK, run_id FK, rule_id FK, passed, observed_value, threshold, sample_rows JSONB)
recon_result(recon_id PK, dataset_id FK, execution_date, check[count|sum|hash],
             left_layer, right_layer, left_value, right_value, delta_pct, passed)
lineage_edge(edge_id PK, upstream_urn, downstream_urn, job, run_id FK)
archival_policy(policy_id PK, dataset_id FK, hot_days, archive_days, purge_after_days, legal_hold)
token_map(dataset_id, column_name, token_key_ref, method[deterministic|dlp])

Design rules you must keep (they come up in interviews constantly): - No credentials in metadata — only Secret Manager references (Phase 12). - pipeline_run is append-only — an audit log you can never UPDATE. - Contracts are versioned, never edited; compatibility enforced on registration (§16). - Every framework component reads only this schema — that's what "parameter-driven" means.

5. Pipeline design (Step 5)

One generated DAG per dataset (one DAG per business process, §7), always this shape:

preflight (contract + source health)
  >> ingest_to_raw            # idempotent: writes raw/.../ingest_date=D/batch_id=
  >> dq_gate_raw              # baseline 5 checks
  >> promote_to_odp           # partition overwrite for date D
  >> promote_to_fdp           # MERGE (SCD2) — idempotent by construction
  >> dq_gate_fdp
  >> build_cdp_products       # dims/facts/rollups, INSERT OVERWRITE partition D
  >> reconcile                # counts/sums across layers
  >> publish_audit_and_lineage

Every task: parameterized by execution_date (never now()), retries 2× with backoff, execution_timeout, and writes a pipeline_run row. Backfill = pipeforge backfill --dataset orders --start 2026-06-01 --end 2026-06-14 (§13: a parameter change, not a code change).

6. Deep-dive commitments (Step 6)

Decisions you will be able to defend at staff level by the end (each is a phase's "Break it" drill):

Topic PipeForge answer
Duplicates 4-layer defense (§18): idempotent producer → ROW_NUMBER in ODP → MERGE in FDP → dedup-proof rollups
Late data event-time partitioning + lookback windows in every incremental job + append-and-compact on RAW (§17)
Schema drift contracts + Schema Registry; breaking change = blocked at registration, not 2 a.m. failure (§16)
Backfill every job partition-parameterized; drill: 14-day backfill in one command (§13)
Skew/scale salting + AQE demo on the clickstream sessionization job (§29)
Failure domains source-down circuit breaker; midway-failure atomic staging swap; downstream freshness alerts (§8)
Cost partition pruning, rollups, serverless, lifecycle tiers; billing export dashboard (§15)

7. Monitoring & SLAs (Step 7)

  • Three pillars (§9): pipeline health (Airflow + pipeline_run), quality (dq_result trends), freshness (MAX(event_ts) checks every 15 min on CDP).
  • SLA published per dataset in the catalog: "orders CDP refreshed by 06:00 with ≥99.5% completeness; 99.9% by noon" (combined SLA, §17).
  • Alert tiers P0–P3 exactly as §9; alerts carry dataset, run, execution_date, runbook link.

9. Milestones & definition of done

Each phase tags a release. Definition of done for v1.0.0: all 11 capabilities demonstrable in a 10-minute live demo: register a brand-new source in the UI → watch the DAG appear → run it → data flows RAW→CDP → DQ scorecard updates → recon passes → lineage graph shows the path → backfill 7 days → purge a customer.

If you can run that demo cold, you can run it in an interview — and no interviewer who has seen it will doubt your seniority.
